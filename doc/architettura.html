Panoramica tecnica per chi riprende o estende il progetto

Stack

LayerTecnologiaNote
FrameworkNext.js 16, App Router, TurbopackTutte le pagine sono client component con dynamic(..., {'{ ssr: false }'}) — SSR disabilitato ovunque (vedi warning sotto)
UIReact 19 + Tailwind CSSNessun component library esterna, stile custom per pagina
DatabaseSupabase (Postgres)Accesso diretto da client browser via @supabase/supabase-js, nessuna API route Next.js
Storage fotoURL esterniNon c'è upload/Supabase Storage: si incollano URL immagine già ospitate altrove
AuthPassword singola hardcodedADMIN_PASSWORD = '1234' in AdminDashboard.tsx — non è autenticazione reale, da sostituire prima di produzione seria
HostingNetlifyDeploy automatico da push su main, plugin Next.js runtime automatico
AnalyticsMicrosoft ClarityScript iniettato in layout.tsx root, tutte le pagine
Nota architetturale importante: non esistono API routes (src/app/api vuoto). Ogni scrittura al database avviene lato client con la chiave anon Supabase. Questo funziona solo perché le tabelle non hanno Row Level Security (RLS) attiva — chiunque col link del sito può leggere/scrivere tutte le tabelle. Va sistemato prima di trattare dati sensibili o multi-cliente.

Duplicazione nota: esistono due pannelli admin paralleli (AdminDashboard e ArtisticDashboard) che scrivono sulle stesse tabelle Supabase con form diversi. È una scelta deliberata per la fase di test/decisione con cliente (vedi changelog.html), ma va consolidato in una versione sola prima del lancio definitivo — altrimenti si rischia divergenza di comportamento tra le due UI.

Convenzioni di codice osservate

  • Ogni pagina app/*/page.tsx è un thin wrapper che importa dinamicamente (next/dynamic, ssr:false) il componente reale da components/
  • Fetch dati sempre in useEffect + funzione async function fetchX() locale al componente, nessun layer di data-fetching condiviso (no React Query/SWR)
  • Form: pattern misto — alcuni usano FormData da e.currentTarget (catturato prima di ogni await, per via del bug di nullificazione di React sugli eventi sintetici), altri usano useState controllato per campo
  • Stile: quasi tutto Tailwind inline, due "temi" cromatici convivono — slate/pink (UI storica) e navy/gold #1a1a2e/#d4af37 (nuova ERP UI)

Debiti tecnici principali (in ordine di priorità)

  1. Sicurezza: password admin hardcoded, nessuna RLS su Supabase → sostituire con Supabase Auth + ruoli (Fase 0 del piano ERP)
  2. Duplicazione admin: consolidare AdminDashboard e ArtisticDashboard in un'unica interfaccia prima del lancio
  3. Dati demo non collegati: Personale, Cast & Audizioni, Materiali, Biglietteria, gran parte di Pianificazione Operativa/Artistica mostrano numeri finti — da collegare a tabelle reali
  4. Campi extra serializzati in testo: dati strutturati (sala, budget, cast assegnato) oggi vivono dentro description come stringa — normalizzare in colonne/tabelle quando si passa a uso reale
  5. Upload immagini: oggi solo URL esterni incollati, nessun upload diretto — aggiungere Supabase Storage se serve caricare foto senza hosting esterno
